# 大屏泛化 + 业务下拉切换 + 酒店样板 — 设计(spec)

- 日期:2026-07-23
- 范围:多业务扩展 4 子项目中的 **子项目 2**(大屏泛化 + 业务下拉切换 + 酒店样板走通 + AI 抽取按业态适配)。
- 状态:已与用户确认设计,待写实现计划。
- 前置:子项目 1(多业务基建)已完成并合并——`business` 增删改、`logo_path`、侧栏「公司经营」业务子菜单、`shop` CRUD、`GET /api/businesses` 均已就绪。

## 1. 背景

子项目 1 打了多业务地基(业务可增删改、侧栏公司经营可按业务切换)。但**数据大屏仍足浴专用**:`dashboard.routes.ts` 的 `KPI_KEYS` 写死 25 个足浴指标,还有 customerStructure(会员/团购/散客)、clockStructure、rechargeStructure、businessStructure(足浴/SPA/小项)、paymentChannels 等足浴专属结构。`Dashboard.vue` 也是足浴手写 ECharts。换业务选不出来、选了也是足浴数据。

AI 抽取(`extraction.service.ts buildMessages`)其实**已 template 驱动**:字段列表来自 `sheet.columns.filter(entry)`,只有 system prompt 写死「你是足浴店财务数据抽取助手」一句。所以抽取适配改动很小。

用户决策(头脑风暴确认):
- **每业务手写 Vue 组件**(②):`FootbathDashboard.vue` 原样不动,新写 `HotelDashboard.vue`;后端也按业务分处理器。加业态前后端各手写一套(用户接受线性成本)。
- 酒店经营报表列**照搬** Excel `2026年07月汉庭酒店报表.xlsx` 的「酒店运营」表(29 列),后续再优化。
- 收入对账 / 费用明细**沿用足浴结构**。
- 「分屏」= 每业务一个独立大屏、侧栏下拉切换(非一屏并排多业务)。

## 2. 目标 / 非目标

### 目标(子项目 2)
- 数据大屏按业务分派:`GET /api/dashboard/overview?businessCode=...`;缺省 `footbath`(向后兼容)。
- 足浴大屏**行为/视觉零改动**(逻辑抽到 `footbath.ts`,加业务过滤后结果不变)。
- 酒店样板走通:酒店模板(经营报表 29 列 + 沿用对账/费用)+ 酒店大屏处理器 + `HotelDashboard.vue`。
- 侧栏「数据大屏」改可展开业务子菜单(复用子项目 1 同款结构),选业务看该业务大屏。
- AI 抽取 system prompt 去硬编码,按业务名参数化。

### 非目标(留给后续)
- 月子会所 / GALA酒吧 / 培训学校 的模板与大屏(子项目 3,各自照酒店范式复制)。
- 酒店收入对账的真实渠道(华住商旅/携程/OTA…)细化——本期沿用足浴对账列。
- adr/occ/revpar 的「总和/总房量」重算——本期用日均,后续优化。
- 「按模板自动选大屏」(免去手写处理器)——本期仍手写分派。

## 3. 架构概览

**按业务分派**:

后端 `/api/dashboard/overview?businessCode=...`:
- 路由解析 `businessCode`(缺省 `footbath`)→ 查 `business` 得 `businessId`(不存在 404)→ 校验 `shopId`(若传)属该业务(否则 400)→ 按 code 分派处理器:`footbath`/`hotel`;其它 → 501「该业务大屏尚未实现」。
- `footbath` 处理器 = 现有逻辑抽到 `dashboard/footbath.ts`;`hotel` 处理器 = 新 `dashboard/hotel.ts`。各处理器 hardcode 本业务的指标→结构映射(② 线性成本)。

前端:
- `Dashboard.vue` 改派发器:取数 + 按 `businessCode` 渲染 `FootbathDashboard.vue` / `HotelDashboard.vue`。
- `FootbathDashboard.vue` = 现 `Dashboard.vue` 图表逻辑原样搬,收 `overview` prop(零视觉改动)。
- `HotelDashboard.vue` = 新手写 ECharts。
- 侧栏「数据大屏」可展开业务子菜单(复用子项目 1 同款)。

数据流:录入(Univer,酒店模板)→ 保存快照 → AI 抽取(prompt 用酒店名 + 酒店模板列)→ `daily_metric`(按酒店 key)→ 酒店大屏处理器按酒店 key 聚合 → `HotelDashboard` 渲染。

## 5. 后端

### 路由 `dashboard.routes.ts`
`GET /api/dashboard/overview?businessCode=footbath|hotel&granularity=...&date=...&shopId=...`
- 新增 `businessCode` 参数,缺省 `'footbath'`(向后兼容)。
- 解析 → 查 `business` 得 `businessId`(不存在 404)→ 校验 `shopId` 属该业务(否则 400)→ 分派。
- 沿用 `computeRange`/`isValidYmd`/`buildTrendSeries` 与 `try/catch→next(e)`。

### `dashboard/footbath.ts`(重构,行为不变)
`computeFootbathOverview({businessId, rangeStart, rangeEnd, trendStart, trendEnd, trendUnit, shopId})`。
- 把现有 `dashboard.routes.ts` 足浴查询逻辑抽来。
- 唯一实质改动:所有 `daily_metric`/`expense` 查询加业务过滤 `shop_id IN (SELECT id FROM shop WHERE business_id=$businessId)`(原全表扫,多业务下必须隔离);`shopRanking` 改该业务内排名。
- 返回 payload 形状**完全不变**(现有 `DashboardOverview` 足浴结构),`FootbathDashboard.vue` 零改动。

### `dashboard/hotel.ts`(新)
`computeHotelOverview({businessId, rangeStart, rangeEnd, trendStart, trendEnd, trendUnit, shopId})`,按酒店 key 聚合,返回 `HotelOverview`(§6)。

### `dashboard.helpers`
抽 `shopFilter(businessId, shopId?)` → `{ sql: 'AND shop_id IN (SELECT id FROM shop WHERE business_id=$B)' [+ 'AND shop_id=$S'], params }`,两处理器共用。

### AI 抽取 `extraction.service.ts`
`buildMessages` 的 system prompt 参数化:传入 `businessName`,「你是`${businessName}`财务数据抽取助手」。调用处(workbook→shop→business.name)取业务名传入。字段匹配逻辑(`sheet.columns.filter(entry)`)不动。

### 校验/错误
`businessCode` 非法 → 404;`shopId` 不属该业务 → 400;业务无处理器 → 501。

## 6. 酒店大屏 payload + 图表

### `HotelOverview` 形状
```ts
{
  granularity, date, rangeStart, rangeEnd, shopId,
  kpis: {
    hotelRevenue, dailyCashTotal, roomRevenue, breakfastRevenue, memberBenefit, // SUM
    overnightRooms, cleanedRooms,                                                // SUM
    adr, occupancy, revpar,                                                       // 日均(AVG over 有数据天)
  },
  revenueTrend:       [{ label, revenue }],                 // hotel_revenue by 粒度桶
  revenueStructure:   { room, breakfast, guestDamage, member, other, offlineRoom },
  roomTypeStructure:  { bigBed, suite, family, superiorBigBed, businessBigBed, superiorTwin, businessTwin },
  channelStructure:   { meituan, ctrip, huazhu, selfuse, walkin },
  expenseBySubject:   [{ subject, amount }],
  shopRanking:        [{ shopId, shopName, revenue }],
}
```

### `HotelDashboard.vue` 图表布局(ECharts,沿用足浴栅格/卡片样式)
1. **KPI 卡行**(10 张):酒店营业收入、日收银总额、客房收入、餐券收入、会员权益、过夜间数、清扫间数(SUM);ADR、OCC、REVPAR(日均,标「日均」)。
2. **营收趋势**折线:`hotelRevenue` by 粒度桶(沿用 `buildTrendSeries` 0 填充)。
3. **收入结构**饼图:客房/餐券/客损/会员权益/其他/线下房。
4. **房型结构**柱图:7 类房型数量(占比由柱高体现)。
5. **渠道房量**饼图:美团/携程/华住/自用免费房/自然客流。
6. **费用科目**柱图 Top10(沿用足浴)。
7. **门店排名**(沿用足浴,按 `hotel_revenue`)。

### 后端聚合规则
- SUM 类:`COALESCE(SUM(numGuard(key)),0)`(沿用)。
- 日均类(adr/occ/revpar):`COALESCE(AVG(numGuard(key)) FILTER (WHERE key 非空非0),0)`——只对有数据天求均,防 0 拉低。后续可优化为「总和/总房量」重算。
- 结构类:各 key SUM,前端按 key 画饼/柱。
- `shopRanking`/`expenseBySubject`/`revenueTrend` 逻辑同足浴,key 换 `hotel_revenue`、业务过滤到酒店门店。

### KPI key 清单(后端 hardcode,② 线性成本)
`hotel_revenue, daily_cash_total, room_revenue, breakfast_revenue, member_benefit, overnight_rooms, cleaned_rooms, adr, occupancy, revpar` + 结构 keys(见上)。
