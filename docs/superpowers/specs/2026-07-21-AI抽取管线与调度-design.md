# AI 数据抽取管线 + 定时调度 - 设计规格

子项目 3。本文件是实现契约:抽取服务、LLM 结构化输出封装、调度器、I-1 数值校验、保存流程改造、前端入口都以本规格为准。

## 1. 背景与问题

v1 的录入归一靠"位置式 sync":用户在 Univer 表按模板列录入 ->「保存并同步」-> 前端 `extractForSync` **按列下标**逐行读 -> 后端 `syncWorkbook` 写 `daily_metric`/`expense`。三个硬伤:

1. **依赖位置**:表头改列、整表转置(原始 Excel 经营报表就是"日期作列、指标作行")、自由表 -> 位置式读取静默错位/漏读,大屏缺数无报错。模板里 `daily_ops` 标了 `layout:'transposed'` 但 `buildFromTemplate` 统一行式建表、`extractForSync` 也只按行读--`layout` 元数据实际未被使用。
2. **无定时保鲜**:必须人工点「保存并同步」才入大屏。员工忘点 -> 大屏停留在旧数据。
3. **I-1 隐患**:`daily_metric.metrics` 是无 schema 单一 JSONB blob,sync 用 `||` 合并、大屏用 26 次 `(metrics->>'key')::numeric` 读(`dashboard.routes.ts:76-106`)。任何非数字字符串溜进数值键 -> 整个大屏接口 500。当前 sync 写入前 `Number()` 兜底所以触发不了,但 AI 抽取作为新写路径必须强制数值校验。

**本子项目用 AI 抽取整体替代位置式 sync**:读工作簿快照 cell -> 调 LLM 按表头关键词**语义化**识别指标 -> I-1 校验 -> 写 `daily_metric`/`expense`。位置式 `extractForSync` + `/sync` + `sync.service` 移除。触发路径三条:保存即抽取(编辑者持锁)、经理手动抽取(manager+ 免锁)、定时调度(当前期每日)。

## 2. 已确认设计决策

1. **替代位置式 sync**:AI 抽取是唯一的入库抽取路径。`extractForSync`(前端)、`POST /workbooks/:id/sync`、`sync.service.ts`/`sync.routes.ts` 移除。校验逻辑(`typeCheck`/`isEmpty`)抽到 `server/src/extraction/validate.ts` 共享复用。
2. **保存即抽取(即时性)**:「保存并同步」按钮 = 存快照(`PUT /snapshot`)+ AI 抽取(LLM)写 PG。保持 v1"保存即进大屏"的即时反馈。持锁编辑者(任意角色)可触发--保存是编辑动作的一部分,不受 §3 的 manager+ 限制。
3. **经理手动抽取(manager+ 免锁)**:独立「AI 抽取」按钮,manager/董事长可点,不要求持锁。基于上次保存的快照抽取,供非编辑态的经理刷新大屏(无需夺锁)。员工不可点此按钮。
4. **接管自动保存不调 LLM**:心跳接管让出前的自动保存只 `persistSnapshot()`(存快照),不触发 AI 抽取--避免自动保存产生 LLM 调用。`performSave` 拆为 `persistSnapshot()` + 抽取两段。
5. **定时调度 = 当前期每日**:默认每天 02:17(`EXTRACT_CRON`),扫当月未软删工作簿,跳过 `updated_at <= last_extracted_at` 的,单并发,每个超时(60s)跳过。`NODE_ENV==='test'` 不启动。
6. **结构化输出 = `response_format: json_object`**:复用 `callDoubao` 的 fetch+Bearer,新建 `callDoubaoJson()`,system prompt 给死 JSON schema,`temperature: 0`。不用 tool-calling(单轮结构化任务)。
7. **调度库 = `node-cron`**(in-process,单实例零新基建)。多实例时加 PG advisory lock(留后续)。
8. **manual_derived 不让 AI 填**:客单价 `customer_price`、对账差额 `mofang_diff` 是人工派生值,AI 只填 `kind==='entry'` 的 35 个字段。
9. **I-1 双保险**:写入侧按模板列 `type` 强制校验(主);大屏读侧 26 处 `::numeric` 加 `CASE WHEN ~ '^-?[0-9]+(\.[0-9]+)?$' THEN ...::numeric ELSE 0 END`(兜底,防历史脏数据)。子项目 2 的 DEFERRED I-1 在此收掉。
10. **溯源轻量标记**:`daily_metric` 加 `last_source TEXT`('ai')与 `last_extracted_at`;`expense` 加 `source TEXT`('ai')。不做 per-key 溯源(改 26 处读侧代价过大,不在本子项目)。`autoSums`(原 sync 返回、前端未用)删除。
11. **抽取模型复用 `DOUBAO_MODEL`**(`doubao-seed-2-1-pro-260628`,长上下文够装一张表);如需独立再加 `EXTRACT_MODEL`。
12. **模板扩展(商品销售等)不在本子项目**:聚焦抽取管线 + 调度 + I-1。

## 3. 抽取流程(核心,唯一入库路径)

```
extractWorkbook(wbId, { source: 'save'|'manual'|'scheduled', userId })
  |
  1. 守卫:工作簿未软删 + 快照存在 + shop/business/template 可解析;未配 key -> {configured:false}
  2. 读 workbook_snapshot.data(Univer IWorkbookData:sheets/sheetOrder/styles)
  3. 对每张模板表(daily_ops / reconciliation / expense):
     a. 按 sheetId 取 sheet(跨重命名稳定,沿用子项目2修复思路)
     b. 序列化:第 0 行表头 + 非空数据行 -> 紧凑 TSV(跳过全空行;daily_ops/reconciliation 日期列空则停;expense 逐行)
        - token 控制:每表上限 400 行,只发非空单元格;预估每表 5-15K tokens
     c. prompt:system(角色+输出 JSON schema+模板列 key->label 映射+规则:只填 entry 列、数值不带单位/千分位、日期 YYYY-MM-DD、不确定留 null) + user(表名+TSV)
     d. callDoubaoJson() -> { rows:[...] } / { expenses:[...] }(见 §4)
  4. I-1 校验(§6.1):逐行逐字段按模板列 type 校验;非法 -> 记 errors,该字段不写;数值清洗(去 ¥/,等)
  5. 写:
     - daily_metric:对每个 date upsert,metrics = 该 date 下所有 entry 键;last_source='ai', last_extracted_at=now()
       ON CONFLICT (shop_id,date) DO UPDATE SET metrics = metrics || $4, last_source='ai', last_extracted_at=now()
     - expense:DELETE WHERE source_workbook_id=$1 再批量 INSERT(source='ai')
  6. 返回报告 { extracted:{dailyMetrics,expenses}, errors:[...], sheets:[{key,rowsIn,rowsOut}], configured:true }
```

**说明**:因 AI 是唯一抽取路径,不存在 sync-vs-AI 冲突。多次抽取(保存/手动/调度)读同一快照、写同一 PG,最后写入胜,幂等可重跑。

## 4. LLM 结构化输出封装

扩展 `server/src/ai/ai.gateway.ts`:
- 新建 `callDoubaoJson(messages)`:fetch `${doubaoBaseUrl}/chat/completions`,body 加 `response_format:{type:'json_object'}`、`temperature:0`、`model:config.doubaoModel`、Bearer `config.doubaoApiKey`。
- 返回 `JSON.parse(choice.content)`;解析失败重试 1 次再抛错。未配 key -> `{configured:false}`。
- 不引入 SDK,沿用现有 fetch 模式(与 chat/poster 一致)。

抽取 JSON schema(system prompt 里描述):
```jsonc
{
  "rows": [ { "date":"2026-07-01", "metrics":{ "revenue":1234.5, "customers_total":56 } } ],  // daily_ops/reconciliation
  "expenses": [ { "pay_date":"2026-07-01","attribution_month":"2026-07","summary":"...","amount":100,"payee":"...","subject":"..." } ]  // expense
}
```
每张表单独一次 LLM 调用(daily_ops / reconciliation / expense 各一轮),按表型裁剪返回结构。一次工作簿最多 3 次 LLM 调用。

## 5. 接口契约

### 5.1 保存(改造现有)
「保存并同步」按钮 -> `performSave()`:
- `persistSnapshot()`:`PUT /api/workbooks/:id/snapshot`(现有,不变;锁校验不变)
- `extractWorkbook()`:`POST /api/workbooks/:id/extract`,body `{ source:'save' }`
- 持锁编辑者可触发(锁已在保存流程校验)

### 5.2 AI 抽取(新)
`POST /api/workbooks/:id/extract`
- 鉴权:`authRequired`;`source='manual'` 时 `requireRole('manager','chairman')`,`source='save'` 时只要持锁(由保存流程保证),`source='scheduled'` 时 system(userId=null)
- auditLog:`ai.extract`,target=workbook id,detail 记 source
- body:`{ source: 'save'|'manual'|'scheduled' }`
- 响应 200:`{ ok:true, configured:true, extracted:{dailyMetrics,expenses}, errors:[{sheetKey,date,key,msg}], sheets:[{key,rowsIn,rowsOut}] }`
- 响应 503:`{ ok:false, configured:false, error:'AI 未配置' }`
- 响应 502:`{ ok:false, error:'调用 AI 失败:...' }`
- 响应 404:工作簿不存在/已软删/无快照
- `source='manual'` 且非 manager+ -> 403

### 5.3 抽取状态
`GET /api/workbooks/:id/extract/status` -> `{ lastExtractedAt, lastSource, lastErrors }`

### 5.4 调度(进程内,无 HTTP)
`extraction.scheduler` 在 `index.ts` 的 `app.listen` 回调启动:`node-cron` 按 `config.extractCron`(默认 `'17 2 * * *'`)跑 `runScheduledExtraction()`:
- `SELECT id, shop_id FROM workbook WHERE period=to_char(now(),'YYYY-MM') AND deleted_at IS NULL`
- 逐个 `extractWorkbook(id,{source:'scheduled',userId:null})`,单并发,失败/超时不阻塞下一个
- `NODE_ENV==='test'` 不启动

### 5.5 移除
- `POST /api/workbooks/:id/sync`(sync.routes.ts)、`sync.service.ts` 删除
- `api.sync`、`extractForSync` 删除

## 6. I-1 数值校验

### 6.1 写入侧(主,extraction/validate.ts)
抽自 `sync.service.ts` 的 `typeCheck`/`isEmpty`:
- 数值列:空->跳过;清洗(去 `¥`、`,` 等)后非数字 -> 记 error 不写;合法 -> `Number()`
- date 列:必须 `/^\d{4}-\d{2}-\d{2}$/`;否则 error
- text 列:原样
- manual_derived 键一律不写

### 6.2 读侧(兜底,dashboard.routes.ts)
26 处 `(metrics->>'key')::numeric` 改:
```sql
COALESCE(SUM(CASE WHEN metrics->>'revenue' ~ '^-?[0-9]+(\.[0-9]+)?$'
                  THEN (metrics->>'revenue')::numeric ELSE 0 END), 0) AS revenue
```
历史脏数据/其他绕过 -> 大屏不 500(脏值当 0)。

## 7. 数据模型改动(schema.sql + migrate)

```sql
ALTER TABLE daily_metric
  ADD COLUMN IF NOT EXISTS last_source TEXT,          -- 'ai'(移除 sync 后唯一值,留字段备查)
  ADD COLUMN IF NOT EXISTS last_extracted_at TIMESTAMPTZ;
ALTER TABLE expense
  ADD COLUMN IF NOT EXISTS source TEXT;               -- 'ai'
```
旧数据(原 sync 写的)字段为 NULL,不影响读侧(读侧只信 metrics 值本身 + I-1 兜底)。

## 8. 前端改动(Workbook.vue + api.ts + types.ts)

- `types.ts`:删 `SyncResult`;加 `ExtractResult { ok; configured; extracted:{dailyMetrics,expenses}; errors:SyncError[]; sheets:{key,rowsIn,rowsOut}[] }`。`SyncError` 保留(复用)。
- `api.ts`:删 `sync`;加 `extractWorkbook(id, source)` -> `POST /api/workbooks/:id/extract`;`extractStatus(id)`。
- `univer.ts`:删 `extractForSync`(及 Workbook.vue 的 import)。`buildFromTemplate` 不变。
- `Workbook.vue`:
  - `performSave` 拆分:
    - `persistSnapshot()` = `endEditingAsync` + `fwb.save()` + `api.putSnapshot`
    - `save()`(「保存并同步」按钮)= `persistSnapshot()` + `api.extractWorkbook(wbId,'save')` -> 显示 `extractResult`(extracted 计数 + errors)
    - 心跳接管让出(`heartbeat` 内)= `persistSnapshot()` only(不抽取)
  - 顶栏加「AI 抽取」按钮:显示条件 `me.role ∈ {manager,chairman}`;点击 -> `api.extractWorkbook(wbId,'manual')` -> 显示报告;不要求 editing/锁;文案"基于上次保存的快照抽取,如需最新请先保存"
  - `syncResult` -> `extractResult`;错误明细沿用现有 `.od-wb-err` 展示
  - 失败:configured:false -> "AI 未配置,请联系管理员";502 -> 提示重试
