# 足浴店财务模块 · 设计文档

- **日期**:2026-07-17
- **项目**:多部门智能经营系统(基于 Univer)
- **子项目**:足浴店财务模块(第一版)
- **状态**:待用户审阅

---

## 1. 概述

### 1.1 目标
基于开源办公 SDK **Univer** 搭建面向小型企业的多部门智能经营系统。Univer 作为统一在线办公底座:在线表格替代零散本地 Excel 文件,多人实时协同编辑。本子项目先落地**足浴店财务模块**(5 家店),并验证可扩展到其他业务的架构。

### 1.2 第一版范围(v1)
1. 协同表格底座(Univer 在线表格 + 多人实时 OT 协同)
2. 数据可视化大屏(5 店对比,可交互)
3. AI 数据分析助手(豆包智能体)
4. 文生图海报模块(豆包,生成每日朋友圈海报)
5. 登录与角色权限 + 操作日志

### 1.3 硬约束
- **可扩展性**:加指标 / 加表 / 加业务 = 改配置,不改代码
- **低学习成本**:员工操作尽量贴近现有 Excel 工作流,不引入新范式
- **便于部署**:华为云,经济型,一键化

### 1.4 技术栈
- 前端:Univer Sheets Preset + React + ECharts + Tailwind
- 后端:Node.js / TypeScript(与 Univer 同构栈,可复用其 Node 运行时解析快照)
- 协同:Univer Server(自部署)
- 数据库:PostgreSQL(RDS)
- AI / 文生图:豆包(字节),公有云 API;DeepSeek 作深度分析备选
- 部署:华为云 ECS + RDS for PostgreSQL + OBS

## 2. 整体架构

采用 **方案 B:干净数据底座 + Univer 录入面 + 自动生成报表/大屏**。核心思想:**录入与展示分离**,PostgreSQL 是衍生视图(报表、大屏、AI)的唯一事实源。

```
                         ┌─────────────────────────────────────────────┐
   浏览器(员工)          │   华为云                                      │
 ┌───────────────────────┴───────────────────────────────────────────┐ │
 │  Univer 在线表格(录入面)   │  数据大屏    │  AI 助手  │  海报模块   │ │
 │  ┌──────────────────────┐ │ ┌────────┐  │ ┌──────┐ │ ┌────────┐ │ │
 │  │ 店铺切换:5 家店       │ │ │ ECharts│  │ │豆包对话│ │ │豆包文生图│ │ │
 │  │ └ 每店工作簿/月        │ │ │ 跨店对比│  │ │分析填表│ │ │每日海报 │ │ │
 │  │   ├ 录入表(人工)      │ │ └────────┘  │ └──────┘ │ └────────┘ │ │
 │  │   └ 台账/派生表(自动)  │ │             │          │            │ │
 │  └──────────────────────┘ │             │          │            │ │
 └─────────┬─────────────────┴──────┬──────┴────┬─────┴─────┬──────┘ │
           │ OT 实时协同(协议)      │ REST API  │ REST API  │ REST    │
           ▼                        ▼           ▼           ▼          │
 ┌────────────────────┐   ┌──────────────────────────────────────┐   │
 │  Univer Server     │   │   应用后端(Node.js/TS)                │   │
 │  (自部署,协同引擎)  │   │  同步服务 │ 大屏API │ AI网关 │ 海报服务 │   │
 │  OT合并/快照/广播    │   │  权限鉴权 │ 操作日志中间件             │   │
 │  快照持久化 ─────────┼──▶│  录入表->PG                            │   │
 └────────────────────┘   └──────────────┬───────────────────────┘   │
          ▲                              ▼                           │
          │ 快照           ┌──────────────────────────┐  ┌────────┐  │
          └───────────────┤  RDS for PostgreSQL       │  │  OBS   │  │
                          │  干净结构化表 + Univer 快照 │  │ 海报图 │  │
                          └──────────────────────────┘  └────────┘  │
                          豆包 API(公有云,HTTPS)◀── AI网关/海报服务  │
 └───────────────────────────────────────────────────────────────────┘
```

**核心要点:**
1. **录入与展示分离**:录入表是源头(员工填),报表表/大屏/AI 从 PG 衍生
2. **PG 存两类数据**:① 干净业务表(给大屏、AI)② Univer 工作簿快照(给协同回放)
3. **可扩展模型**:`业务 -> 门店 -> 模块`,新业务注册模板即可复用全部基础设施
4. **登录鉴权 + 操作日志**为横切模块,贯穿所有业务模块

## 3. 数据模型

### 3.1 元数据表(可扩展骨架)

```
business(业务)      id, code, name              例:足浴 / 餐饮 / 零售
  └ shop(门店)      id, business_id, code, name     例:大河坎店等 5 家
  └ template(模板)  id, business_id, version, definition(JSONB)   ← 扩展性核心
  └ workbook(工作簿) id, shop_id, period(YYYY-MM), univer_unit_id, template_version
```

**`template.definition`**(JSONB,版本化)是系统理解表格的"说明书":
```jsonc
{
  "sheets": [
    { "key": "daily_ops", "label": "经营报表",
      "layout": "transposed",            // 转置:日期作列、指标作行
      "columns": [
        {"key":"date","label":"日期","type":"date","identity":true},
        {"key":"revenue","label":"营业收入","type":"number","unit":"元","kind":"entry"},
        {"key":"customers_total","label":"总客流","type":"int","kind":"entry"},
        {"key":"customer_price","label":"客单价","type":"number","kind":"manual_derived"},
        {"key":"revenue_total","label":"合计","type":"number","kind":"auto_sum"}
        // kind: entry=人工录入 | manual_derived=人工填(员工自写函数/值) | auto_sum=后端自动求和
        // 加指标 = 这里加一行,不改任何代码
      ],
      "grain": "per_day"
    },
    { "key": "reconciliation", "label": "收入对账", "layout": "row_per_day", "grain": "per_day" },
    { "key": "expense", "label": "费用明细", "layout": "row_per_transaction", "grain": "per_transaction" }
  ]
}
```

### 3.2 业务事实表(指标用 JSONB 保灵活)

```
daily_metric   id, shop_id, date, business_code,
               metrics JSONB,            ← {revenue:8026.66, customers_total:25, ...}
               source_workbook_id, updated_at
expense        id, shop_id, pay_date, attribution_month, summary, amount,
               payee, subject, extra JSONB, source_workbook_id, source_row
```

**指标存 `daily_metric.metrics` 的 JSONB**:新增指标 = 新增 JSON key,**不改表结构、不迁移**。配 GIN 索引按 key 查询。

### 3.3 权限与日志表

```
user           id, username, password_hash, name, role, department,
               phone, last_login_ip, last_login_at, status,
               created_by, created_at
operation_log  id, user_id, user_name, ip, action, target,
               detail, result, created_at
```
- `role`:枚举(chairman=董事长 / manager=经理 / employee=员工)
- `department`:字段或独立表(本期按字段处理)

### 3.4 录入表 -> PG 同步(模板驱动,不写死)
1. **触发**:工作簿快照保存时(协同中持续产生,做防抖),后端用 Univer Node 运行时解析快照
2. **解析**:按 `template.definition` 的列定义读取录入表结构化区域,逐行映射到 `daily_metric` / `expense`
3. **校验**:按列类型校验;错误单元格在 Univer 里**标红 + 提示**
4. **写入**:upsert 进 PG -> 触发报表重算

因"按模板描述符解析"而非"写死第几列是什么",模板一改同步自动适配。

### 3.5 报表/资金台账(自动生成,只读)
资金台账(滚动结余)由后端从 `daily_metric`(收入)+ `expense`(支出)按日累加算出,写回 Univer 只读视图表。员工看到的报表自动生成,无需维护跨表公式,杜绝 `#REF!` 类错误。

## 4. Univer 集成(工作簿组织、协同、同步)

### 4.1 工作簿组织(员工视角,零新概念)
- 登录后看到**店铺列表**(5 家足浴店;未来加新业务/新店即列表多一项)
- 点某店 -> 进入**该店当月工作簿**(默认当月,可切历史月)
- 工作簿内的表:

| 表 | 录入方式 | 说明 |
|---|---|---|
| 经营报表(转置) | **人工填** | 每日经营指标;原始数据 + 派生值(客单价等)人工填;**仅"合计"行后端自动求和** |
| 收入对账 | **人工填** | 支付渠道(抖音/美团/POS/支付宝/微信)+ 摩方对账,外部数据 |
| 费用明细 | **人工填** | 一笔一行,逐笔费用 |
| 资金台账(滚动结余) | **自动生成,只读** | 由收入 + 支出按日累加算出 |

员工操作 = 填格子,和 Excel 无异。无"提交表单""配置组件"等新概念。

### 4.2 协同(Univer Server,自部署)
- 自部署 Univer Server 处理 OT 实时协同:多人同改同店录入表,冲突自动合并,光标/选区互见
- 工作簿快照持久化到 PG
- **待核实**:Univer Server 开源/Pro 授权边界(见 §13 风险)

### 4.3 扩展性落地
新建某店某月工作簿时,系统按 `template.definition` **自动生成录入表的列结构**(表头、类型、校验)。模板改了,下月工作簿自动套新模板;历史月保留旧版本(工作簿绑 `template_version`,老数据不坏)。

## 5. 数据可视化大屏

### 5.1 技术选型
React + **ECharts**(图表/联动/钻取原生支持)+ **Tailwind**(玻璃拟态/动效)。**不使用 OpenDesign**,前端由本项目自行设计实现。数据来自后端大屏 API(读 PG 聚合),同步后近实时刷新。

### 5.2 视觉风格("科技驾驶舱"质感)
- **深色主题**:深空藏蓝底(`#0B1220`)+ 微渐变
- **玻璃拟态卡片**:半透明 + 毛玻璃模糊 + 细青色描边微光,圆角 12px,柔和阴影
- **配色**:主青 `#22D3EE`(营收/主线)、金 `#FBBF24`(任务目标)、5 店各一色;预警色 绿(达标)/黄(观望)/红(落后)
- **动效**:KPI 数字计数滚动、折线渐变填充、hover 发光、切换丝滑过渡
- **字体层级**:KPI 大号粗体,标签清晰,留白透气;16:9 自适应栅格

### 5.3 主题可切换
用**设计令牌(CSS 变量)**统一管理配色,预设多套主题(深空科技暗色 / 商务浅色 / 暖橙等),顶栏一键切换,所有卡片与图表同步换肤。

### 5.4 交互(点哪看哪,层层下钻)
- **店铺钻取**:点 5 店排名某家 -> 整屏切到该店详情,↩ 返回总览
- **KPI 卡点击**:钻取该指标趋势详情
- **图表联动**:点柱/线/饼块 -> 其他图表同步过滤
- **时间**:"月/周/日"切换 + ◄► 翻月 + 日期选择
- **对比模式**:选 2 家店并排对比
- **预警**:任务进度 vs 时间进度,落后标红 + AI 简报点名
- **hover**:tooltip 明细;图例点击显隐
- **自动刷新**:同步后刷新 + 手动 🔄

## 6. AI 数据分析助手(豆包智能体)

### 6.1 选型
- **主用豆包**(字节):与海报模块统一供应商;支持 function calling / 智能体;国内合规、延迟低
- **深度分析备选 DeepSeek**:复杂归因/预测时调用
- 后端封装 **AI 网关**,模型可后续切换,业务不动

### 6.2 能力(员工大白话用,零学习成本)
- **数据问答**:问数据 -> AI 调工具查 PG -> 回答(可带图表)
- **经营分析**:异常预警、归因、趋势、店间对比(大屏"AI 经营简报"每日自动生成)
- **辅助办公**:自然语言帮填表、生成报表文字说明、起草对账说明、解释指标算法

### 6.3 架构(智能体 + 工具调用,均为本项目自研)
```
员工:"大河坎为什么落后?" ──自然语言──▶ AI 网关(按权限过滤上下文)
                                          │ 组装:店铺/月份/权限 + 模板定义
                                          ▼
                                    豆包(智能体)──工具调用──▶ query_metrics
                                          │           ──▶ query_expenses
                                          │           ──▶ compare_shops  (均读 PG)
                                          │◀──结构化结果──────────────┘
                                          ▼
                             自然语言回答 + 图表数据 ──▶ 对话面板 / 大屏简报
```
- **智能体编排逻辑**:本项目自研(接对话 -> 决定调哪些工具 -> 执行 -> 组织回答)
- **工具调用函数**:本项目自研(`query_metrics` / `query_expenses` / `compare_shops` 等,本质是查 PG 的函数,按模板 metric key 通用查询)
- 豆包仅提供大模型 + function calling 能力
- 工具函数通用、模板驱动 -> 新指标/新业务自动可查,不改提示词、不写新工具(满足扩展性)

### 6.4 数据合规
采用 **(a) 豆包公有云 API**:最简单、最便宜,**数据出域到字节**(用户已确认接受)。备选:(b) 豆包专属/私有化(不出域,成本高);(c) 本地开源模型(完全不出,需 GPU)。

### 6.5 内容可见性(本期)
因本期**不区分角色内容可见性**,AI 网关目前**不按店铺过滤数据**;待内容权限启用后再开。

## 7. 文生图海报模块(豆包)

### 7.1 选型与策略
- **豆包文生图**(字节,即梦/Seedream)生成主视觉背景
- **图文分离**:文生图只生成背景主视觉;**文字层(店名、日期、文案、二维码)用服务端 Canvas 叠加**,保证文字清晰、数据准确、可编辑

### 7.2 海报内容(提示词驱动,不取数)
- **不从 PG 自动取数**,与财务数据完全解耦
- 内容来源:**用户输入提示词**(描述主题/促销/风格/文案)+ **手动设置日期**
- 可选:AI(豆包文生文)帮把简短提示扩写为完整文案,或用户自写
- 店名/二维码按所选店铺带出(店铺基础信息,不算取数)

### 7.3 生成流程
```
选店铺 + 选模板风格
      ▼
输入提示词(主题/促销/风格)+ 手动设日期   ← 内容全在这里,不连 PG
      ▼
(可选)AI 扩写文案
      ▼
豆包文生图 -> 主视觉背景
      ▼
服务端 Canvas 叠加文字层(店名/日期/文案/二维码)
      ▼
预览 -> 微调 -> 导出(朋友圈竖版 1080×1920 / 方图)-> 存 OBS -> 下载/分享
```

### 7.4 模板与风格(可复用、可扩展)
- 海报模板配置驱动:布局区(标题/主视觉/数据/二维码)、配色、字体
- 风格预设:足浴养生风、节日风、促销风(各对应不同文生图提示词)
- 5 店各有 logo/店名/二维码,模板共用;新业务换模板+提示词即可复用

### 7.5 低学习成本
一键生成:选店 -> 写提示词 -> 设日期 -> 出图 -> 下载。员工不用设计技能;不满意一键重生成;支持批量给 5 店各出一张。

## 8. 登录与角色权限 + 操作日志

### 8.1 角色与权限模型

| 角色 | 用户管理权 | 内容可见性 | 操作日志 |
|---|---|---|---|
| **董事长** | 增删改查**所有经理 + 员工**;可创建经理 | 不区分 | 查看全部 |
| **经理** | 增添/管理**本部门 / 自己创建的员工** | 不区分 | 查看本部门 |
| **员工** | 无;只能登录使用 | 不区分 | 不查看 |

规则:
- **员工账号由经理(或董事长)创建后才能登录**,无自注册
- **经理由董事长创建**;经理**不能管其他经理、不能创建经理**
- **本期不区分内容可见性**(所有人登录看同样数据);后续可加按角色/店铺的内容权限

### 8.2 员工信息字段
姓名、用户名(账号)、密码(bcrypt 哈希)、角色、**部门**、手机号、**最近登录 IP**、最近登录时间、状态(启用/禁用)、创建人、创建时间。

### 8.3 登录
- 账号 + 密码;密码哈希存储(bcrypt)
- 登录时记录 **IP + 时间**,签发 token,登录态贯穿所有模块
- 禁用账号禁止登录;失败次数限制防爆破

### 8.4 操作日志(审计)
- 记录**重要活动 + 操作人**
- 记录项:登录/登出、用户增删改/启停、模板修改、海报生成、重要配置变更、**数据录入/编辑(按操作粒度)**、数据删除/批量操作
- **数据录入/编辑按操作粒度记**(如"编辑了大河坎店 2026-07 经营报表"),不记每个单元格(单元格级历史在 Univer 快照里已有,避免日志爆炸)
- 字段:时间、操作人、IP、操作类型、对象、详情、结果(成功/失败)
- 后端中间件统一记录,跨所有模块

### 8.5 集成
登录鉴权 + 操作日志为**横切模块**接入应用后端,所有模块(Univer 前端、大屏、AI、海报)基于登录态;操作日志中间件统一拦截记录。

## 9. 可扩展性总结

**一个模板描述符(`template.definition`)驱动所有环节**:

```
template.definition (JSONB, 版本化)
   ├─▶ Univer 录入表结构(表头/类型/校验)自动生成
   ├─▶ 同步服务(按列定义解析 -> PG,不写死)
   ├─▶ 报表/资金台账自动生成
   ├─▶ 大屏组件(配置选 metric key 即可绘图)
   ├─▶ AI 上下文(自动认识新指标,不改提示词)
   └─▶ 海报模板(布局/提示词预设,新业务换模板即可)

指标存 JSONB -> 加指标不改表、不迁移
business -> shop -> module -> 新业务复用全部基础设施
```

| 场景 | 操作 | 改代码? |
|---|---|---|
| 加一个指标 | 模板 definition 加一行列定义 | ❌ |
| 加一张录入表 | 模板加一个 sheet 定义 | ❌ |
| 改指标类型/单位 | 改 definition 对应字段 | ❌ |
| 接入新业务 | 新建 business + 一套 template,挂门店 | ❌ |
| 大屏展示新指标 | 大屏组件配置选这个 key | ❌ |
| AI 认识新指标 | AI 读 template definition 作上下文 | ❌ |

代价:模板需有人维护(管理员通过模板编辑界面改,或本期先用 JSON 配置)。

## 10. 部署(华为云,经济型)

```
                    用户浏览器(HTTPS)
                         │
                    ┌────▼─────┐
                    │  Nginx   │ 反向代理 + TLS
                    └────┬─────┘
        ┌────────────────┼─────────────────┐
        ▼                ▼                 ▼
 ┌────────────┐  ┌──────────────┐  ┌──────────────┐
 │Univer Server│  │ 应用后端(Node)│  │  (poster 图) │
 │ OT协同/快照  │  │ 同步/大屏API/ │  └──────┬───────┘
 │             │  │ AI网关/海报/  │         │
 │             │  │ 鉴权/日志     │         │
 └──────┬──────┘  └──────┬───────┘         │
        │                │                 │
        └────────┬───────┘                 │
                 ▼                          ▼
        ┌──────────────────┐        ┌──────────────┐
        │ RDS PostgreSQL    │        │     OBS      │
        │ 业务表+Univer快照  │        │ 海报图存储    │
        │ (自动备份)         │        └──────────────┘
        └──────────────────┘
                 │
        豆包 API(公有云,HTTPS 出域)◀── AI网关/海报服务调用
```

- **1 台 ECS**(4-8 核 / 8-16G):Nginx + Univer Server + 后端,用 **docker-compose 一键起**,环境变量配置
- **RDS for PostgreSQL**:托管、自动备份
- **OBS**:存海报图
- **域名 + HTTPS**(华为云免费证书)
- 部署文档化:步骤、环境变量、备份恢复写清楚,方便迁移/重装

## 11. 错误处理与数据质量
- **录入校验**:按模板类型校验,错误单元格在 Univer 里标红 + 提示
- **同步**:失败重试 + 日志;大屏显示"最后同步时间",异常可见
- **AI**:基于工具查 PG,不瞎编数据;模型失败优雅降级(提示稍后重试)
- **豆包 API**:失败重试,海报/AI 都有兜底
- **数据库**:RDS 自动备份 + 定期验证恢复

## 13. 风险与待确认
1. **Univer Server 开源/Pro 授权边界** -- 需核对官方文档,可能影响是否需商业授权(影响成本)。落地前确认。

## 14. 关键决策记录
| 决策点 | 选择 | 理由 |
|---|---|---|
| 架构方案 | 方案 B(干净底座 + Univer 录入面 + 自动报表) | 根治 Excel 脆弱性,可扩展,保表格协同体验 |
| 录入设计 | 录入归一;经营报表人工填(仅合计自动求和);收入对账/费用明细人工;资金台账自动生成 | 低学习成本 + 最大人工控制权,消除跨表重复录入 |
| 指标存储 | JSONB | 加指标不改表不迁移(换扩展性) |
| 大屏 | React+ECharts+Tailwind,可交互+主题切换,不用 OpenDesign | 自主可控视觉,交互原生支持 |
| AI 选型 | 豆包公有云(数据出域接受)+ DeepSeek 备选 | 统一供应商、低成本 |
| AI 架构 | 自研智能体 + 工具调用 | 控制 PG 工具/权限/扩展性 |
| 海报内容 | 提示词驱动 + 手动设日期,不取 PG | 与财务解耦,灵活;文生图背景 + Canvas 叠加文字 |
| 权限 | 董事长/经理/员工;经理只管本部门员工;本期不区分内容可见性 | 满足用户管理需求,后续可扩展内容权限 |
| 操作日志 | 记重要活动 + 数据录入/编辑(操作粒度) | 审计全覆盖,控制日志体积 |
| 部署 | 华为云 1 ECS(docker-compose)+ RDS + OBS | 经济型、一键部署 |
